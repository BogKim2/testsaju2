# Orchestrator — 핵심 조율자 설계

## Orchestrator란?
> 에이전트 시스템의 **두뇌**. 사용자 요청을 분석하고, 계획을 세우고, 올바른 에이전트에게 일을 시키고, 결과를 취합한다.

단순한 "LLM에 질문하기"가 아니라, **여러 전문 에이전트를 지휘하는 감독자**이다.

---

## Orchestrator의 3대 책임

```
┌──────────────────────────────────────────────────────────┐
│                      ORCHESTRATOR                         │
│                                                           │
│   ① PLAN           ② MEMORY          ③ ROUTE             │
│   ────────         ────────          ────────             │
│   무엇을 할지      무엇을 알고       누가 할지             │
│   계획 수립        있는지 기억       에이전트 배분          │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

## ① Planner (계획 수립 모듈)

### 역할
사용자의 복잡한 요청을 **실행 가능한 단계들**로 분해한다.

### 작동 방식

```
사용자 요청:
  "하네스 설계 파일을 검토하고 문제점 리포트를 만들어줘"

        ↓ Planner 분석

실행 계획 (Plan):
  Step 1: [File Parser 도구] DXF 파일 파싱
  Step 2: [Design Agent]    설계 구조 파악
  Step 3: [BOM Agent]       BOM 추출 및 검증
  Step 4: [Spec Agent]      전기 스펙 검사
  Step 5: [Quality Agent]   품질 규칙 체크
  Step 6: [LLM]             결과 종합 → 보고서 작성
```

### 계획 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **순차 실행** | 이전 결과가 다음 입력 | BOM → 스펙 검증 → 보고서 |
| **병렬 실행** | 독립된 작업 동시 처리 | 설계 검토 + BOM 검토 동시 |
| **조건부 실행** | 결과에 따라 경로 분기 | 오류 발견 시 → 상세 분석 |

### Planner 의사결정 로직

```python
# Planner가 계획 수립 시 고려하는 것들

def create_plan(user_request):
    # 1. 요청 유형 파악
    request_type = classify_request(user_request)
    #   → "설계 검토", "BOM 분석", "경로 최적화" 등

    # 2. 필요한 에이전트 목록 결정
    required_agents = select_agents(request_type)

    # 3. 실행 순서 결정
    execution_order = sort_by_dependency(required_agents)

    # 4. 각 단계에 도구 할당
    plan = assign_tools(execution_order)

    return plan
```

## ② Memory Manager (기억 관리 모듈)

### 두 가지 기억

```
┌─────────────────────────────────────────────────────┐
│                   Memory Manager                     │
│                                                      │
│  ┌─────────────────────┐  ┌───────────────────────┐ │
│  │    Short-term       │  │     Long-term          │ │
│  │    단기 기억         │  │     장기 기억           │ │
│  │                     │  │                        │ │
│  │  · 현재 대화 컨텍스트│  │  · 설계 이력           │ │
│  │  · 진행 중인 작업    │  │  · 검증 결과 DB        │ │
│  │  · 임시 계산 결과    │  │  · 표준 규칙 라이브러리 │ │
│  │  · 세션 동안만 유지  │  │  · 부품 스펙 데이터     │ │
│  │                     │  │  · 영구 저장           │ │
│  │  → Redis (빠름)      │  │  → SQLite / Vector DB │ │
│  └─────────────────────┘  └───────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### Memory가 없으면 생기는 문제

```
❌ Memory 없음:
사용자: "A커넥터 스펙 확인해줘"
AI:     "A커넥터는 12핀, 최대 10A입니다"

사용자: "그럼 이 커넥터를 B하네스에 적용해도 돼?"
AI:     "무슨 커넥터를 말씀하시는 건가요?" ← 기억 못함!

✅ Memory 있음:
사용자: "A커넥터 스펙 확인해줘"
AI:     "A커넥터는 12핀, 최대 10A입니다"

사용자: "그럼 이 커넥터를 B하네스에 적용해도 돼?"
AI:     "앞서 확인한 A커넥터(12핀, 10A)를 B하네스에
         적용 시, B하네스의 최대 전류(8A)와 비교하면
         문제 없습니다" ← 이전 대화를 기억함!
```

### Vector Memory (문서 검색)

```
저장되는 것:
  - 과거 설계 검토 보고서
  - 표준 규격 문서 (KSC, ISO, 현대모비스 기준 등)
  - 부품 데이터시트

검색 방식:
  "12핀 방수 커넥터" 검색
    → 유사한 과거 설계 이력 검색
    → 관련 표준 규격 자동 참조
    → 유사 사례 기반 추천
```

## ③ Task Router (작업 라우터 모듈)

### 역할
어떤 에이전트가 이 작업을 처리할지 **결정하고 배분**한다.

### 라우팅 규칙

```
입력 요청 분석
      │
      ├─ 파일 파싱 필요? ──────────────→ File Parser Tool
      │
      ├─ "BOM", "부품", "재료" 키워드? → BOM Agent
      │
      ├─ "설계", "도면", "레이아웃"?  → Design Agent
      │
      ├─ "전압", "전류", "저항"?      → Spec Agent
      │
      ├─ "품질", "검사", "규격"?      → Quality Agent
      │
      └─ "경로", "라우팅", "배선"?    → Routing Agent
```

### 멀티 에이전트 조율

```
복잡한 요청 → 여러 에이전트 동시 실행

Orchestrator
  ├── [병렬] Design Agent   → "레이아웃 분석 완료"
  ├── [병렬] BOM Agent      → "부품 목록 추출 완료"
  └── [병렬] Spec Agent     → "스펙 데이터 수집 완료"
       │
       └── [순차] Quality Agent  → "세 결과를 모아 검사"
            │
            └── [순차] LLM → "최종 보고서 작성"
```

## Orchestrator 상태 머신

```
[IDLE]
  │ 사용자 요청 수신
  ▼
[PLANNING]       ← Planner가 계획 수립
  │ 계획 완성
  ▼
[DISPATCHING]    ← Task Router가 에이전트 배분
  │ 실행 시작
  ▼
[EXECUTING]      ← 에이전트들이 작업 수행
  │ 결과 수집
  ▼
[AGGREGATING]    ← 결과 취합 및 최종 정리
  │ 보고서 완성
  ▼
[RESPONDING]     ← 사용자에게 응답 반환
  │ 완료
  ▼
[IDLE]
```

## 오류 처리 전략

| 상황 | 대응 |
|------|------|
| 에이전트 실패 | 재시도 최대 3회 → 실패 시 빈 결과({}) 반환, 하위 에이전트 SKIP 처리 |
| LLM 타임아웃 | 30초 후 타임아웃 → 부분 결과 반환 |
| 파일 파싱 오류 | 오류 원인 분석 후 사용자에게 안내 |
| 계획 수립 실패 | 기본 템플릿 계획으로 폴백 |
